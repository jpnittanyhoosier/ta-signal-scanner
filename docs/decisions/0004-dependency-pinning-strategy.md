# ADR 0004: Version-Range Pinning Without Lockfile

**Status:** Accepted
**Date:** 2026-04-21
**Deciders:** Project owner

---

## Context

When declaring Python dependencies, three strategies exist for controlling which versions actually get installed:

- **Option A: Unpinned / loose bounds** — declare `fastapi>=0.110` with no upper bound. Always installs the latest compatible version. Fragile: a breaking release in any dependency or transitive dependency can silently break the build.

- **Option B: Version ranges with upper bounds** — declare `fastapi>=0.110,<0.120`. Allows bug-fix and minor updates but blocks major-version breaks. Still non-deterministic across installs (two installs minutes apart can pick different patch versions), but bounded.

- **Option C: Loose bounds + lockfile** — declare loose bounds in `pyproject.toml`, but commit a separate `requirements.lock` (or `requirements-lock.txt`) generated by a tool like `pip-compile`. Lockfile contains exact versions and hashes. Fully reproducible. Requires a regeneration step when updating dependencies.

The project is currently single-developer, single-machine. The future AWS deployment will require greater reproducibility (multiple environments must match exactly).

---

## Decision

Adopt **Option B: version ranges with upper bounds** for v1.

For each dependency, declare a lower bound at a known-working version and an upper bound at the next major version (or the next minor version for libraries with looser semver discipline).

Example:
```toml
"fastapi>=0.110,<0.120",     # major-version bound
"yfinance>=0.2.40,<0.3",     # minor-version bound (yfinance breaks more often)
```

Plan to **revisit and likely upgrade to Option C** when the AWS migration requires reproducible build artifacts.

---

## Alternatives Considered

### Alternative A: Unpinned / loose bounds

**Rejected because:** This is a foot-gun. A breaking release in any direct or transitive dependency can break a fresh `pip install` with no code change on the project's part. The cost of debugging such breakage far exceeds the cost of typing upper bounds.

### Alternative C: Loose bounds + lockfile

**Considered seriously, deferred:**

Genuinely the professional standard for production systems. Provides:
- Bit-for-bit reproducible installs across machines
- A clear "upgrade event" (regenerate the lockfile, review the diff, commit)
- Resistance to supply-chain attacks (hashes verify package integrity)

Deferred for v1 because:
- Single developer, single machine — the reproducibility-across-machines benefit does not yet apply
- Adds tool complexity (`pip-tools` or `uv` for lockfile generation)
- Adds a workflow step (lockfile regeneration) that adds friction to dependency changes
- The risk Option C primarily mitigates (transitive dependency drift breaking a clean install) is rare for a small dependency set on a developer machine

The AWS migration phase is the natural point to upgrade. At that point, multiple environments (developer machine, CI, Lambda artifact) must produce identical behavior, and the lockfile becomes mandatory rather than optional.

---

## Consequences

### Positive

- Dependency declaration is simple: one file, no separate lockfile to maintain.
- Bug-fix releases of dependencies are picked up automatically on `pip install`.
- Major-version breaks are blocked by upper bounds, preventing the worst class of "it broke and I don't know why" incidents.
- Lower friction for trying out new versions: bump the upper bound, see what happens.

### Negative

- Two `pip install` runs minutes apart on the same `pyproject.toml` may install different patch versions. Acceptable for a single-developer project; would not be acceptable for a multi-environment production system.
- No protection against supply-chain attacks via package hash verification. Acceptable risk for a personal project pulling from PyPI; would be revisited for a public-facing or commercial deployment.
- When the AWS migration arrives, there will be a non-trivial task to introduce lockfile tooling and re-pin dependencies at exact versions.

### Follow-up Required

- Write a follow-up ADR during AWS migration phase that supersedes this one and adopts Option C.
- At that future ADR, also re-evaluate whether to migrate from `pip` to `uv` for lockfile generation (`uv pip compile` is faster and integrated).
